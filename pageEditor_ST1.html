Splash Screen

The Splash Screen is the first thing users see. I wanted to make the wait feel friendly and intentional, so I coded a custom loader with just HTML and CSS (no images or scripts required). The loader spins smoothly thanks to @keyframes in CSS, and I made sure the color contrast is sharp for all users. For structure, I used a simple <div> container with an <h1> greeting. Everything is lightweight for a fast load.
Tags & Techniques: <div>, <h1>, @keyframes, animation, blue and white color palette.

Volunteer Page

I built the volunteer page using semantic HTML, with every opportunity as an <article> inside a <section>. Ratings are done with <span> and CSS for a star effect—no JS needed. For layout, I used display: flex and flex-wrap to keep cards tidy, even on mobile. All form fields have <label>s, and I checked color contrast for accessibility.
Tags & Techniques: <section>, <article>, display: flex, form with label, custom CSS for rating stars and responsive layout.

Content Page

This page is all about structure. I used lots of <section> and <article> tags for each topic, plus in-page links using <a href="#section"> for easy jumping. The blue accent and spacing match the other pages. Headings are nested logically for both users and screen readers. I also checked that every alt tag is meaningful for images. CSS media queries make the layout work on phones and laptops.
Tags & Techniques: <section>, <article>, <nav>, font-size and line-height for readability, media queries for responsiveness.

Challenges and Lessons Learned

  • Making things truly responsive: Some layouts broke on mobile, so I learned to test constantly and use flex-wrap and min-width tricks.
  • Accessibility details: I forgot some label tags at first, but fixing this helped me see why semantic HTML is so important for everyone.
  • CSS organization: At first, styles got messy. Switching to CSS variables and grouping properties by section made things much easier to manage.
  • Lesson: Simplicity wins. The more semantic and lean my code, the better it worked—for both users and validators.

Accessibility

  • Images have descriptive alt tags for screen readers.
  • All forms use <label> and support keyboard navigation.
  • Color choices pass contrast checks (tested with WAVE and browser tools).
  • Navigation and headings are logical for both users and assistive tech.
  • Pages are usable with just a keyboard and work with high-contrast mode.

Compliance

  • All HTML and CSS validated with the W3C validator (see Reference 2).
  • No personal data collected, matching student web guidelines and Janet requirements (see Reference 1).
  • Accessibility and privacy are considered, including WCAG 2.1 principles where practical.
  • No inline CSS or scripts, to maintain clarity and standards.
